# 完整项目分析蓝图 (Project Blueprint)

## 项目概要

一个基于 Spring Boot 与 LangChain4j 的智能代码分析平台。其核心能力是利用 RAG（检索增强生成）与 AI 代理技术，对用户项目源码进行自动化分析、审计、问答与可视化，旨在辅助开发人员快速理解项目架构与代码逻辑。

## 1. 技术栈与框架

| 层级 | 技术/框架 | 说明 |
| :--- | :--- | :--- |
| **基础框架** | Spring Boot 3.x | 应用主体框架，提供依赖注入、Web 服务等核心能力。 |
| **AI 集成框架** | LangChain4j | 用于集成大语言模型 (LLM)、构建 AI 链 (Chain) 与智能体 (Agent) 的核心框架。 |
| **向量存储** | LangChain4j 内置 / 本地文件 | 用于存储代码片段 Embedding，支持语义检索。具体实现（如 InMemory / Chroma）**待补充验证**。 |
| **前端技术** | Tailwind CSS, PostCSS | 用于构建前端界面的样式工具，暗示存在一个 Web 前端。其具体框架（如 React, Vue）**待补充验证**。 |
| **并发处理** | `ExecutorService` | 用于管理异步任务与并发处理的线程池。 |

## 3. 核心业务逻辑与数据流

### 3.1 项目加载与索引构建流程
1.  **触发**: 用户上传或指定项目路径。
2.  **抓取**: `CodeCrawlerService` 与 `FileCrawlerService` 遍历项目文件。
3.  **处理**: `IndexService` 与 `ProjectIndexManager` 对源码进行分块、解析。
4.  **向量化**: 通过 Embedding 模型将代码块转换为向量，存入向量数据库。
5.  **元数据生成**: 同时构建项目结构地图 (`ProjectMap`)，并可能持久化为 `Project-Map.json`。
6.  **就绪**: 项目索引状态被更新，可供后续查询与分析。

### 3.2 RAG 智能问答流程
1.  **接收请求**: `AnalysisController` 接收用户自然语言查询。
2.  **混合检索**: `RagChatService` 并行执行：
    *   **向量检索**: 在代码片段向量库中进行语义相似度搜索。
    *   **关键词检索**: 进行传统文本匹配作为补充。
3.  **上下文构建**: 合并、去重、排序检索结果，并截断至 `maxContextChars` 限制。
4.  **对话记忆集成**: 关联当前会话的历史消息 (`maxMemoryTurns`)。
5.  **AI 推理**: 将整合后的上下文、历史及问题发送至配置的 DeepSeek 模型。
6.  **流式响应**: 通过 Server-Sent Events (SSE) 实时返回生成结果给前端。

### 3.3 AI 代理审计流程
1.  **触发**: 用户请求对特定代码进行深度审计。
2.  **代理调度**: `AuditAssistant` 作为协调者，可能将任务分解。其 `agentic` 模式实现细节**待补充验证**。
3.  **多维度分析**: 调用 `LogicAuditService` 等专业服务进行针对性检查。
4.  **报告生成**: 汇总分析结果，构建结构化的 `AuditReport`。
5.  **可视化辅助**: `ProjectMapService` 可能被调用以生成架构图。

## 4. 关键设计模式与架构思想

| 模式/思想 | 应用点 | 说明 |
| :--- | :--- | :--- |
| **门面模式 (Facade)** | `AnalysisController` | 为复杂的后端分析服务群提供一个统一的、简化的 HTTP 接口。 |
| **策略模式 (Strategy)** | `RagChatService` 中的检索器 | 向量检索与关键词检索作为可互换的检索策略。 |
| **代理模式 (Proxy)** | `AuditAssistant` | 作为 AI 代理的访问点，控制对底层复杂 AI 审计逻辑的访问，可添加超时 (`agenticAuditTimeoutMs`) 等控制逻辑。 |
| **工厂模式 (Factory)** | `ProjectIndexManager` | 负责协调和组织整个项目索引生命周期的构建，是索引相关组件的工厂与管理器。 |
| **配置化驱动** | 所有服务的参数配置 | 如 `vectorTopK`, `minScore` 等，使系统行为高度可调，适应不同场景。 |
| **响应式流** | SSE 流式输出 | 对长时 AI 生成任务采用 Server-Sent Events，提升用户体验。 |

## 5. 配置体系与关键参数

| 类别 | 参数示例 | 默认值 | 作用 |
| :--- | :--- | :--- | :--- |
| **RAG检索** | `vectorTopK`, `keywordTopK`, `minScore` | 4, 2, 0.35 | 控制检索的广度、精度与召回率。 |
| **上下文管理** | `maxContextChars`, `maxMemoryTurns` | 8000, 4 | 防止 Prompt 过长，平衡历史记忆与性能。 |
| **AI代理审计** | `enableAgenticAudit`, `agenticAuditTimeoutMs` | false, 8000ms | 控制是否启用高级代理模式及执行超时。 |
| **流式传输** | `sseTimeoutMs`, `sseHeartbeatMs` | 600000ms, 15000ms | 维持长连接稳定性，防止中间件超时断开。 |

## 6. 已识别的技术风险与待验证点

| 风险点 | 级别 | 分析与缓解建议 |
| :--- | :--- | :--- |
| **内存与资源泄漏** | 高 | 大量项目索引、向量数据驻留内存；`ExecutorService` 异步任务管理不当。需验证是否有索引卸载机制及线程池的优雅关闭。 |
| **超时与容错** | 中 | AI 服务调用、复杂审计流程存在超时 (`agenticAuditTimeoutMs`) 配置，但全局异常处理与降级策略**待补充验证**。 |
| **向量检索性能** | 中 | 项目规模增长后，纯内存或简单文件向量库的检索效率可能成为瓶颈。需确认向量库选型及是否有优化策略。 |
| **配置复杂性** | 低 | 大量配置参数提高了灵活性，也增加了运维复杂度。需要清晰的文档或配置管理界面。 |
| **前端与后端协同** | 待补充 | 前端具体技术栈、API 契约、SSE 连接管理方式**待补充验证**。 |
| **审计逻辑深度** | 待补充 | `LogicAuditService`, `InteractiveAuditService` 的具体审计规则和 AI 提示词工程细节**待补充验证**，这是核心业务价值所在。 |

---
**蓝图总结**: 该项目是一个设计良好的、以 **RAG 为核心**的垂直领域 AI 应用。架构清晰，采用 **Spring Boot + LangChain4j** 的成熟组合，通过**混合检索**与**配置化**策略平衡了精度与灵活性。其亮点在于将 AI 问答、自动化审计、架构可视化流程进行了**服务化整合**。当前主要风险集中于**资源管理**与**大规模数据下的性能**，核心业务逻辑（审计规则）的细节有待进一步探查。
